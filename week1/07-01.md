# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 
    - webserver 구현/index.html 반환응답/원초적인 HTTP 성공 응답과정과 파싱
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 없음 

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 
    - url 파서 클래스를 두어 static method로 파싱하게 구현, 각 확장자마다 content-type을 다르게 반환하도록 구현 
- 그렇게 판단한 이유: 
    - webserver와 handler 클래스는 파싱의 역할을 하는 것은 옳지 않다고 판단하였습니다. 즉, 책임이 다르다고 느꼈기에 분리하였습니다. 또한 static 메서드로 메모리를 절약하고 각 확장자마다 contnet type을 설정하여 여러 요청에도 유연한 대응이 가능하게 했습니다. 

- 고려했지만 선택하지 않은 대안: 
    - Enum 클래스로 각 content-type을 관리하려했으나 이렇게 될시 점점 톰캣의 디스패처 서블릿처럼 프로젝트가 매우 커지고 기간 내 할 수 없을 거같다고 느꼈습니다.

- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 
    - 객체지향, 유지보수 면에서는 효율적이지 못합니다. 다만 MVP 면에서는 효율적이라 생각합니다. 

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 리소스 url 지정이 계속 실패
- 처음 추측한 원인: 리소스 위치 . 이 resources로 지정되지 않느다. 
- 원인을 확인하기 위해 시도한 방법: 로그롤 출력, chrome의 개발자도구로 네트워크 확인 
- 실제 원인: 스프링이 기본으로 resources 쪽으로 잡아주는거에 당연하다고 생각해서 문제였습니다. 즉, 고정관념.. 
- 해결 방법: 직접 ./resources/static으로 url 고정
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 실제 resources url을 먼저 찍어보고 판단 

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: chrome 네트워크 확인, ai 최대한 안쓰고 여러 방식으로 접근해서 원인 분석
- 그 접근이 도움이 된 이유: 빠르게 http응답으로 잘못된 응답인지 확인 가능 , ai에 의존하지 않았기에 정해진 방식보다 다른 방식으로 접근가능
- 다음에도 반복하고 싶은 점: 로그를 계속 잘 확인할 것입니다. 여러 방면에서 접근해보겠습니다. 

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: url에 /../ 접근을 막지 못함 
- 그렇게 된 원인: 자원을 파싱하는데 해당 /../config 와 같은 파일 접근이 가능해짐. 
- 다음에는 다르게 해볼 구체적인 행동: 원천적으로 /../와 같은 상위 디렉터리 접근 패턴을 등록해서 원천차단
