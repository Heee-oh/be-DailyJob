### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 각 요청 처리 Handler를 싱글톤으로 사용하기 위해 HandlerMapping을 구현하여 싱글톤으로 빠르게 요청 url에 대하여 핸들러 반환 
- 그렇게 판단한 이유: 매 요청마다 핸들러를 생성하는 것이 메모리 면에서 매우 비효율적이라고 판단 
- 고려했지만 선택하지 않은 대안: 리플렉션으로 애노테이션을 직접 구현
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 프로젝트가 매우 커지면서 복잡해져 제대로 완성할 수 없었을 것이다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: JUnit의 @Test와 AssertJ를 사용할 수 없었고, 이후에는 Failed to load JUnit Platform 오류로 테스트가 실행되지 않았다.
- 처음 추측한 원인: Spring 의존성이 없거나 JUnit import가 잘못됐다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 테스트 파일 위치와 build.gradle의 의존성을 확인하고, Gradle 테스트 런타임 클래스패스를 조회했다.
- 실제 원인: 테스트 파일이 처음에는 src/main/java에 있었고, 이동 후에는 Gradle 9에서 필요한 junit-platform-launcher가 런타임 클래스패스에 없었다.
- 해결 방법: 테스트 파일을 src/test/java로 이동하고 junit-platform-launcher를 testRuntimeOnly 의존성으로 추가했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 파일이 올바른 Source Set에 있는지, 컴파일 의존성과 런타임 의존성이 모두 구성됐는지 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 테스트 코드 작성하여 확인 , ai로 분석 및 취약점 분석 
- 그 접근이 도움이 된 이유: 테스트 코드로 빠르게 문제점을 파악할 수 있었음, ai가 오타 등의 문제를 잡아줌
- 다음에도 반복하고 싶은 점: 작업 완료후 ai로 분석을 항상 시도, 비지니스로직은 단위테스트 항상 작성 

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 단위 테스트가 통과한 것을 보고 전체 WAS가 완성됐다고 판단했다. 실제 HTTP 요청에서는 404·403 응답 생성 실패와 쿼리가 붙은 정적 파일의 라우팅 오류가 남아 있었다.

- 그렇게 된 원인: 로그인과 회원가입의 정상 흐름에 집중했고, 파서부터 응답 출력까지 이어지는 통합 경로와 예외 경로를 충분히 검증하지 않았다.

- 다음에는 다르게 해볼 구체적인 행동: 완료하기 전에 200, 400, 401, 403, 404, 500 응답을 직접 요청하는 체크리스트를 만든다. 정적 파일에 쿼리가 붙는 경우, 없는 파일, 잘못된 URL, 경로 탐색 요청도 curl로 확인한다. 테스트마다 정적 Database 상태도 초기화해 실행 순서에 의존하지 않게 만든다.
