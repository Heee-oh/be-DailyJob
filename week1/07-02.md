
# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 
    - ContentType 을 ENUM으로 관리
    - /../ 와 같은 상위 디렉토리 접근 방지
    - 쿼리파라미터 존재 여부에 따른 확장자 추출
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - 절차 지향적인 코드를 객체 지향적으로 변경하기를 못하였습니다.  

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: ContentType을 ENUM으로 관리함
- 그렇게 판단한 이유: ContentType은 예측불가능하지 않고 한정되어있으며 HTTP 프로토콜에서 필수적으로 사용되기에 재사용 가능하면서 상수로 하나의 클래스 내에 관리할 수 있는 ENUM으로 수정하였습니다. 
- 고려했지만 선택하지 않은 대안: 없음
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 없음 

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 확장자가 계속 null로 나와 예외가 발생
- 처음 추측한 원인: 확장자 추출을 잘못해서 계속 실패하는 것으로 판단 
- 원인을 확인하기 위해 시도한 방법: 확장자 추출 부근에 log를 찍어서 확인 
- 실제 원인: 확장자를 초기 null로 초기화하고 핸들러 분기가 확장자 존재시와 쿼리 파라미터가 있을때를 제외한 나머지 즉, /home 과 같은 요청시 확장자가 null이였기 떄문이였다. 
- 해결 방법: 확장자가 핸들러 분기를 다 통과해도 null이면 index.html요청으로 판단하여 ext 를 html로 변경 
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: null에 대한 무분별한 사용을 금하고 실행 흐름을 제대로 확인하면서 개발

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 터미널에서 /../와같은 상위 파일 접근 시도 
- 그 접근이 도움이 된 이유: 브라우저의 경우 /../ 접근은 방지하기에 그냥 지나칠 수 있었지만 공격자 입장에서 ai에게 상위 파일 접근 방법에 대해서 질문하여 해당 방법을 알아내 공격시도를 해보았다. 이를 통해서 경로 검증이 필요하다는 것을 알게 되었고 이를 구현하여 더 안전하게 구현할 수 있었다.
- 다음에도 반복하고 싶은 점: 내가 찾지 못하는 문제점이나 취약점을 ai에게 검증받아서 실제로 테스트를 해보기 

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 로깅으로 계속 체크하지만 원인을 분석하지 못함 
- 그렇게 된 원인: 제대로 된 실행 로직 분석을 하지않았고, 테스트 코드를 작성하지 않음 
- 다음에는 다르게 해볼 구체적인 행동: 천천히 실행 로직을 따라가면서 구현하고 테스트코드를 작성
