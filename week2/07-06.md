### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: HTTP 파싱을 시작라인, 헤더, 바디로 나눠 책임 분산

- 그렇게 판단한 이유: 하나의 파서가 요청 라인, 헤더, body를 모두 처리하면 책임이 커지고 새로운 요청 형식을 추가하기 어렵다고 판단했다.

- 고려했지만 선택하지 않은 대안: 메서드<url, 핸들러>
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: url을 처음 키로 잡아서 해당 엔드포인트에 대한 여러 메서드들을 적용할 수있지만 반대로 잘못된 url에 잘못된 메서드 사용시 404응답을 반환한다. 
보통은 405로 반환한다고 하여 매우 고민이다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: postman 요청시 2번 요청이 나가며, 한번은 null, 한번은 올바른 요청이 나가는 현상
- 처음 추측한 원인:
    HttpLineReader가 요청 데이터를 잘못 소비하거나 Postman이 HTTP 요청을 두 번 전송한다고 추측했다.

- 확인을 위해 시도한 방법:
    스레드별 request line과 클라이언트 연결 정보를 로그로 출력했다.

- 확인한 내용:
    HTTP 요청이 두 번 전달된 것이 아니라 TCP 연결이 두 개 생성됐고, 실제 HTTP 요청은 한 연결에서만 전달됐
    다. Postman이 연결을 두 개 생성한 정확한 내부 이유는 확인하지 못했다.

- 해결 방법:
    request line이 null이면 잘못된 HTTP 요청으로 응답하지 않고 EOFException으로 구분해 조용히 연결을 종료
    했다. 이미 종료된 소켓에 응답을 작성하면서 발생하던 Broken pipe도 방지했다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
    추측만으로 코드를 변경하지 않고 스레드별 로그를 추가해 빈 연결과 실제 POST 요청을 구분했다.

- 도움이 된 이유:
    문제의 원인이 body 파서나. HttpLineReader가 요청을 잃어버린 것이 아니라, 서로 다른 TCP 연결이었다는 사
    실을 확인할 수 있었다.

- 다음에도 반복하고 싶은 점: 
    네트워크 문제를 확인할 때 요청 단위뿐 아니라 연결, 스레드, 클라이언트 포트까지 함께 기록하고 판단하겠
    다.
