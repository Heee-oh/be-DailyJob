Ai 활용
Di와 ApplicationContext를 webserver 하위에서 상위로 뽑아내는 리팩터링 과정에서 발생한 DI 컨테이너 버그 3개를 연쇄적으로 해결하고, 마지막에 네이밍을 정리했습니다.

1. default 패키지 문제 (분석만)

- 증상: 상위로 옮긴 ApplicationContext를 다른 클래스들이 import 못 함
- 원인: 파일을 src/main/java/ 바로 아래(= 무명/default 패키지)에 둠 → 자바에서 named 패키지는 default 패키지 클래스를 import 불가
- 해결 방향 제시: 전용 패키지 부여 → 이후 di 패키지로 정리됨

2. 스캔이 깊은 패키지에 도달 못 함 (분석 → 수정)

- 증상: DispatcherHandler 빈이 없다(BeanNotFoundException)
- 원인: scan() 재귀가 부모 경로를 안 붙이고 디렉토리 "이름"만 넘김 → webserver.handler 같은 2단계 이상 패키지 미스캔
- 이 과정에서 파생된 선행 슬래시 문제(빈 base + / → getResource null)와 슬래시/점 구분자 충돌도 분석

3. IllegalName / NoClassDefFoundError (수정)

- 증상: NoClassDefFoundError: IllegalName: webserver/handler.StaticHttpHandler
- 원인: loadClass에 슬래시+점 혼합 이름 전달
- 수정: 로딩 시점에만 resourcePath.replace('/', '.')로 변환

4. InvocationTargetException (수정)

- 증상: 생성자 내부 NPE가 감싸져 올라옴 (리플렉션 실패 아님)
- 원인: register가 Map.put의 반환값(이전 값=null)을 그대로 리턴 → 의존성이 null로 주입 → HandlerMapping의 null::create에서 NPE
- 수정: addBean 후 실제 생성한 빈을 반환하도록 변경
